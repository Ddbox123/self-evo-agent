# SPEC 开发流程模板

**版本：** v1.2
**日期：** 2026-04-19
**用途：** 虾宝自我进化系统的标准化开发流程

---

## 概述

### 目的

本模板为虾宝自我进化系统提供标准化的开发流程，确保每次开发任务都有完整的需求分析、设计、实现、测试、文档归档和质量门控。

### 适用范围

- 所有 Phase 1-8 的功能开发
- Bug 修复和优化
- 重构和模块拆分
- 提示词相关变更

### 核心原则

1. **Phase 递进**：每轮开发完成后，Phase 状态递增
2. **规划-实施-归档**：规划文档和任务报告分离存放，供 AI 查询
3. **强制质量门控**：测试覆盖、代码审查、提示词打靶（如涉及）
4. **诚实标注状态**：✅ 完整 / ⚠️ 框架 / ❌ 未实现
5. **Core First 原则**：优先使用 core 中封装好的功能，agent.py 只负责运行循环逻辑
6. **INDEX.md 联动**：每次文件更新必须同步 INDEX.md

---

## Core First 开发准则

> **⚠️ 重要**：agent.py 是运行循环逻辑的入口，**不是**功能堆积的地方。

### 开发检查流程

```
需求 → 检查 core/ 是否已有功能
         │
         ├─ 有 → 直接导入使用
         │         └─ agent.py 只写调用代码
         │
         └─ 无 → 在 core/ 对应类别添加功能
                   │
                   ├─ 1. 创建/修改 core/{category}/xxx.py
                   ├─ 2. 添加功能实现
                   ├─ 3. 更新 core/{category}/README.md
                   ├─ 4. 在 core/{category}/__init__.py 导出
                   └─ 5. agent.py 导入使用
```

### Core 模块说明文档要求

每个 core/ 子目录必须包含 `README.md` 文件，说明：
- 目录功能概述
- 包含的模块列表
- 主要类和函数说明
- 使用示例

### 违规警告

- ❌ **禁止**：在 agent.py 中添加未封装的功能逻辑
- ❌ **禁止**：跳过 core 直接在 agent.py 实现功能
- ❌ **禁止**：更新 core 模块后不更新 INDEX.md
- ✅ **必须**：先检查 core，再开发，最后导入 agent.py

### 提示词系统架构（2026-04-21 更新）

> 本节描述当前提示词组件体系，所有开发任务必须遵循。

**核心文件职责划分：**

| 文件 | 职责 | 行数目标 |
|------|------|---------|
| `core/core_prompt/SOUL.md` | 身份定义、核心铁律、思维格式、禁止行为 | <120 行 |
| `core/core_prompt/AGENTS.md` | 完整 SOP 操作流程、协议、工具速查 | <500 行 |
| `requirement/SPEC/SPEC_Agent.md` | 开发规范与质量门控（本文件） | 模板性质 |
| `workspace/prompts/DYNAMIC.md` | 本世代动态任务描述（Agent 自行维护） | 动态 |
| `workspace/prompts/IDENTITY.md` | Agent 身份定义 | 动态 |
| `workspace/prompts/USER.md` | 用户信息与环境 | 动态 |

**组件优先级（数字越小越靠前）：**

```
SOUL(10) → TASK_CHECKLIST(20) → CODEBASE_MAP(30) → DYNAMIC(40) →
IDENTITY(50) → AGENTS(60) → SPEC(65) → USER(70) →
MEMORY(80) → current_rules(85) → TOOLS_INDEX(90) → ENV_INFO(100)
```

**任务编排流程（<plan> 标签驱动）：**

```
模型输出 <plan> 文本块
    ↓
agent.py: PlanOrchestrator.extract_plan_tag()  提取
    ↓
agent.py: PlanOrchestrator.parse_and_store()  解析并存储
    ↓
TaskPlanner (core/orchestration/task_planner.py)  持久化
    ↓
PromptManager: TASK_CHECKLIST 组件  渲染 Markdown 清单
    ↓
注入系统提示词 → 引导模型逐任务执行
```

**响应解析标签（ResponseParser 支持）：**

| 标签 | 提取内容 | 应用位置 |
|------|---------|---------|
| `<think>...</think>` | 思考过程 | UI 显示、日志 |
| `<plan>...</plan>` | 任务计划 | TaskPlanner 存储 |
| `<active_components>...</active_components>` | 组件切换 | PromptManager |
| `<tool_call>...</tool_call>` | 工具调用 | 工具执行 |

**开发约束：**
- ❌ 禁止将 SOP 操作流程写入 SOUL.md（保持 SOUL 精简）
- ❌ 禁止将身份/使命信息写入 AGENTS.md（保持 AGENTS 为流程参考）
- ✅ 提示词变更后必须运行 `python tests/prompt_debugger.py --suite` 验证

### INDEX.md 版本号更新规则

| 场景 | 版本号变更 | 示例 |

## 质量门控

### 代码审查清单

- [ ] **命名规范**：模块文件 snake_case，测试文件 test_*.py，类 PascalCase，函数 snake_case

### 7.2 改进触发条件

| 触发条件 | 优先级 | 说明 |
|----------|--------|------|
| 某步骤频繁出现问题 | P1 | 必须改进 |
| 某步骤耗时超出预期 50% | P2 | 建议改进 |
| AI 发现更好的方法 | P3 | 可选改进 |
| 遇到反复出现的同类问题 | P1 | 必须改进 |

### 7.3 模板更新规则

当复盘发现需要改进时，由 AI 工具决定是否更新本模板：

**必须更新模板的情况：**
- 某流程步骤被反复跳过或忽略
- 发现系统性遗漏（如总是忘记更新 INDEX.md）
- 质量门控存在漏洞

**更新流程：**
1. AI 在复盘时提出改进建议
2. 下轮开发前，由 AI 判断是否采纳
3. 如采纳，更新本模板并记录到变更日志
4. 如不采纳，在复盘记录中说明原因

### 7.4 流程健康度指标

| 指标 | 定义 | 目标值 | 监控方式 |
|------|------|--------|----------|
| 规划准确率 | 实际交付与规划的符合度 | >= 90% | 复盘记录统计 |
| 测试覆盖率 | 有测试的模块占比 | >= 80% | pytest --cov |
| 文档完备率 | 按时提交报告的比例 | 100% | report_history/ 检查 |
| Bug 遗留率 | 上线后发现的 Bug 数 | <= 5% | 后续追踪 |
| 流程偏差率 | 实际流程与模板的偏差 | <= 10% | 复盘自评 |

### 7.5 复盘记录归档

**归档位置：**
- `requirement/SPEC开发流程/复盘记录/`

**命名格式：**
- `review_{YYYYMMDD}_Phase{N}_{任务摘要}.md`

**示例：**
- `review_20260418_Phase8_探索引擎实现.md`
