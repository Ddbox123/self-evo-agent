## 二、流程优化方案

### 优化原则

```
┌─────────────────────────────────────────────────────────────┐
│                     优化三原则                                │
├─────────────────────────────────────────────────────────────┤
│  1. 简化不删质：减少流程负担，保持质量门控                       │
│  2. 小任务快办：简单任务走轻量流程                             │
│  3. 大任务拆办：复杂任务保持完整流程                           │
└─────────────────────────────────────────────────────────────┘
```

---

### 优化后的双轨流程

#### 轻量流程（≤1小时的小任务）

```
需求 → 简单设计 → 实现 → 测试 → Git提交
        ↓
   INDEX.md 增量更新（非全量同步）
```

**适用场景：**
- Bug 修复
- 简单功能添加
- 文档修正
- 少量代码重构

**检查清单（简化版）：**
```
[ ] 理解需求
[ ] 检查 core/ 是否有相似功能
[ ] 实现
[ ] 简单测试
[ ] Git 提交 + 简短 commit message
[ ] 增量更新 INDEX.md（仅修改日志）
```

---

#### 标准流程（复杂任务）

```
Phase N: 任务名称
├── 阶段0：开发前准备
│   ├── 完成检查清单
│   ├── 写规划文档（可选，>4小时的任务必须）
│   └── 确认无阻塞依赖
├── 阶段1：需求分析
├── 阶段2：设计（决策记录）
├── 阶段3：实现（Core First 检查）
├── 阶段4：测试（覆盖率 >= 80%）
├── 阶段5：文档归档
└── 阶段6：Git 提交
```

**适用场景：**
- 新增核心模块
- 大规模重构
- Phase 级别任务
- >4小时的任务

---

### INDEX.md 联动优化

#### 当前问题

每次更新需要同步多个章节，容易遗漏：
- "当前实现状态"表格
- "项目结构"目录树
- "修改日志"
- "测试统计"
- "规划文档索引"
- "任务完成报告索引"

#### 优化方案：增量联动

| 更新类型 | 必须更新的章节 | 可延迟的章节 |
|---------|---------------|-------------|
| 新增 core 模块 | 1. 修改日志<br>2. 核心模块状态表 | 项目结构目录树（下季度整理） |
| 修改 core 模块 | 1. 修改日志<br>2. 核心模块状态表 | - |
| Phase 完成 | 1. Phase 状态表<br>2. 修改日志<br>3. 实施路线图 | - |
| Bug 修复 | 1. 修改日志 | 无 |

**原则：**
- 强制：修改日志（所有变更必须记录）
- 强制：状态表（只更新涉及的模块）
- 可选：目录树（每季度整理一次）

---

### 文档优化

#### 当前问题

文档过多，维护成本高：
- 规划文档
- 任务报告
- 复盘记录
- 模块 README
- INDEX.md

#### 优化方案：精简为"两文一表"

| 文档类型 | 位置 | 内容 | 何时写 |
|---------|------|------|--------|
| **规划文档** | `requirement/task_{name}.md` | 设计方案、决策记录、预期交付物 | >4小时的任务 |
| **任务报告** | `report_history/task_{name}.md` | 实际完成情况、遇到的问题、后续计划 | 重要任务 |
| **修改日志** | `INDEX.md` 末尾 | 所有变更的增量记录 | 所有变更 |

**删除：**
- ❌ phase 复盘记录（合并到任务报告）
- ❌ 单独的模块 README.md（用 __init__.py 的 docstring 代替）

**保留：**
- ✅ SPEC_Agent.md（流程模板，模板性质不变）
- ✅ INDEX.md（全局索引 + 修改日志）

## 三、Core First 强化方案

### 当前问题

agent.py ~1070行，远超 500行目标。说明 Core First 原则未真正落地。

#### 根因分析

```
agent.py 问题：
├── 直接实现了功能逻辑（应该在 core/）
├── 状态管理散落各处
├── 缺乏清晰的分层架构
└── 缺少明确的职责边界
```

### 强化规则

#### 规则1：agent.py 硬性约束

```
┌─────────────────────────────────────────────────────────────┐
│  agent.py 只能包含：                                          │
├─────────────────────────────────────────────────────────────┤
│  1. think_and_act()      - 核心思考-行动循环                  │
│  2. run_loop()           - 主循环入口                         │
│  3. _invoke_llm()        - LLM 调用                          │
│  4. 状态初始化           - 必要的单例初始化                    │
│                                                             │
│  禁止出现：                                                  │
│  ❌ 业务逻辑（如任务分类、优先级计算）                          │
│  ❌ 工具执行逻辑                                             │
│  ❌ 复杂的条件判断                                           │
└─────────────────────────────────────────────────────────────┘
```

#### 规则2：Core 模块分类强化

```
core/
├── infrastructure/     # 基础设施（工具、状态、事件）
├── orchestration/      # 编排（LLM、记忆、任务）
├── decision/          # 决策（决策树、优先级、策略）
├── evolution/         # 进化引擎
├── knowledge/         # 知识系统
├── learning/         # 学习系统
├── autonomous/       # ⚠️ 自主探索（Phase 8 待完成）
├── capabilities/     # 能力系统（提示词、任务管理）
├── ecosystem/        # 工具生态（Skill系统）
└── logging/         # 日志系统
```

#### 规则3：新增功能检查流程

```
需求 → 检查 core/ 是否有相似功能
         │
         ├─ 有 → 直接导入
         │         └─ agent.py 只写调用代码（<10行）
         │
         └─ 无 → 在 core/ 对应类别添加
                   │
                   ├─ 1. 创建/修改 core/{category}/xxx.py
                   ├─ 2. 在 __init__.py 导出
                   ├─ 3. 实现功能
                   ├─ 4. 写 docstring
                   └─ 5. agent.py 导入使用
```

#### Step 3: 迁移 agent.py

## 六、测试优化

### 当前问题

- 测试文件分散
- 部分测试无实际断言
- 覆盖率统计不清晰

### 优化方案

#### 1. 简化测试分类

| 测试类型 | 目录 | 覆盖率要求 |
|---------|------|-----------|
| 单元测试 | `tests/unit/` | >= 80% |
| 集成测试 | `tests/integration/` | 核心流程通过 |
| 冒烟测试 | `tests/smoke/` | 快速验证 |

#### 2. 快速测试命令

```bash
# 冒烟测试（每次提交前必须）
pytest tests/smoke/ -v

# 单元测试
pytest tests/unit/ -v --cov=core --cov-fail-under=80

# 完整测试（发布前）
pytest tests/ -v --cov=core --cov=tools --cov-fail-under=80
```

#### 3. 提示词打靶

```bash
# 每次修改 SOUL.md / AGENTS.md 后必须
python tests/prompt_debugger.py --suite
```
